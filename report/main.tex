\begin{abstract}
Frequent itemset mining is central to association-rule learning, but dense transaction datasets make classical algorithms expensive because the search space grows exponentially and support counting repeatedly touches the database. This project compares the classical Apriori algorithm with a modern BitTable-inspired Tensor-Eclat implementation. The contemporary direction is motivated by 2022 onward work on parallel BitTable and Spark-based frequent itemset mining, while the implemented prototype maps vertical bitsets to vectorized NumPy/PyTorch tensor operations. Two optimization strategies are studied: vertical transaction-id representations that remove repeated horizontal scans, and parallel bitwise support counting that replaces subset checks with tensor intersections. The evaluation uses the Chess, Connect, and Accidents benchmark settings required in the project brief, across minimum support thresholds from 0.9 to 0.6. Results show that the optimized Tensor-Eclat approach reduces execution time by up to 108$\times$ and peak memory by up to 29.7$\times$ in the reported benchmark artifact. The study confirms that representation choice is the dominant practical factor in dense frequent itemset mining: Apriori remains clear and theoretically important, but vertical bitsets and parallel intersections are more scalable for large correlated datasets.
\end{abstract}

\section{Introduction}
Frequent itemset mining (FIM) discovers sets of items that co-occur above a minimum support threshold. These itemsets are then used to generate association rules for retail baskets, clinical records, fraud analysis, clickstreams, and bioinformatics. Although the problem statement is compact, the number of possible itemsets is exponential in the number of distinct items. Dense datasets such as Chess and Connect are especially difficult because many long combinations remain frequent.

Apriori is the standard baseline because it introduced a simple but powerful downward-closure rule: if an itemset is frequent, all of its subsets must also be frequent \cite{agrawal1994fast}. This property prunes many impossible candidates, but Apriori still performs repeated database scans and can generate a very large number of candidates. Modern work therefore shifts the bottleneck from repeated scanning to compact representations and parallel execution. Recent BitTable and Spark-based studies use compact bit structures and distributed processing to reduce scan and shuffle overhead \cite{kittipron2022bittable,albana2022hybrid,fernandezbasso2024spark}.

This project makes three contributions. First, it implements an Apriori baseline and a Tensor-Eclat alternative over the same transaction format. Second, it applies two concrete optimizations: vertical tidset/bitset representation and vectorized tensor intersections. Third, it reports time, memory, candidate count, frequent itemset count, and speedup over multiple support thresholds using the benchmark artifact in the repository.

\section{Literature Review}
Agrawal and Srikant introduced Apriori and established the anti-monotonic pruning principle used by most later FIM algorithms \cite{agrawal1994fast}. Its limitation is not correctness but cost: each itemset level requires support counting over candidate sets, so runtime grows quickly when support decreases.

Zaki's Eclat changes the representation from horizontal transactions to vertical transaction-id lists \cite{zaki2000scalable}. In Eclat, support counting is performed by intersecting tidsets, which is often faster than scanning all transactions for every candidate. FP-Growth, proposed by Han, Pei, and Yin, removes candidate generation by compressing the database into an FP-tree \cite{han2000mining}. These classical alternatives show that data representation is as important as pruning strategy.

Recent post-2022 work emphasizes parallel and big-data execution. Kittipron and Jaruskulchai proposed a BitTable-based parallel approach on Spark, using a compact data structure and MapReduce-style processing \cite{kittipron2022bittable}. Al-Bana et al. presented a Spark-based hybrid FIM approach for large datasets, targeting Apriori's repeated scan cost \cite{albana2022hybrid}. Fernandez-Basso et al. later proposed Spark versions of Apriori, Apriori-TID, Eclat, and association-rule mining, highlighting that distributed in-memory processing can overcome memory and runtime limits of classical single-machine algorithms \cite{fernandezbasso2024spark}.

The research gap for this project is practical and implementation-focused: many modern algorithms require distributed clusters, while student hardware often has a strong local GPU. The implemented Tensor-Eclat prototype therefore keeps the Eclat/BitTable idea but maps bitset intersections to vectorized tensor operations.

\section{Algorithms}
\subsection{Classical Apriori}
Apriori performs breadth-first mining. At level $k$, it joins frequent $(k-1)$-itemsets to form candidate $k$-itemsets, prunes candidates whose subsets are not frequent, scans the database, and retains candidates whose support exceeds $\sigma$.

\begin{algorithm}[htbp]
\caption{Apriori Baseline}
\begin{algorithmic}[1]
\REQUIRE Transaction database $D$, support threshold $\sigma$
\ENSURE Frequent itemsets $F$
\STATE $L_1 \leftarrow$ frequent 1-itemsets in $D$
\STATE $k \leftarrow 2$
\WHILE{$L_{k-1} \neq \emptyset$}
    \STATE $C_k \leftarrow$ join and prune $L_{k-1}$
    \FORALL{$t \in D$}
        \FORALL{$c \in C_k$}
            \IF{$c \subseteq t$}
                \STATE increment support$(c)$
            \ENDIF
        \ENDFOR
    \ENDFOR
    \STATE $L_k \leftarrow \{c \in C_k: support(c)\geq\sigma |D|\}$
    \STATE $k \leftarrow k + 1$
\ENDWHILE
\RETURN $\cup_k L_k$
\end{algorithmic}
\end{algorithm}

If $C_k$ is the number of candidates at level $k$, $|D|$ is the number of transactions, and $w$ is average transaction width, the support-counting time is approximately $O(|D|\sum_k C_k w)$. Space is dominated by the largest candidate/frequent level.

\subsection{Tensor-Eclat}
Tensor-Eclat uses vertical representation. Each item maps to a Boolean vector of transaction membership. A candidate support count becomes the population count of an intersection:
\begin{equation}
support(X \cup Y)=|TID(X)\cap TID(Y)|.
\end{equation}
The implementation uses PyTorch tensors when available and NumPy Boolean arrays otherwise. Items are sorted by ascending support to reduce the DFS suffix search tree early.

\begin{algorithm}[htbp]
\caption{Tensor-Eclat with Vertical Bitsets}
\begin{algorithmic}[1]
\REQUIRE Transaction database $D$, support threshold $\sigma$
\ENSURE Frequent itemsets $F$
\STATE Build vertical bitset $B_i$ for each item $i$
\STATE Keep items with $sum(B_i)\geq\sigma |D|$
\STATE Sort items by increasing support
\FORALL{item $i$ in sorted items}
    \STATE DFS($\{i\}, B_i$, suffix after $i$)
\ENDFOR
\STATE
\STATE \textbf{procedure} DFS(prefix, prefix\_bits, suffix)
    \STATE $S \leftarrow \emptyset$
    \FORALL{$(j,B_j)$ in suffix}
        \STATE $B_{new}\leftarrow prefix\_bits$ AND $B_j$
        \IF{$sum(B_{new})\geq\sigma |D|$}
            \STATE add $prefix\cup\{j\}$ to $F$
            \STATE append $(j,B_{new})$ to $S$
        \ENDIF
    \ENDFOR
    \FORALL{$(j,B_j)$ in $S$}
        \STATE DFS($prefix\cup\{j\},B_j$, remaining suffix)
    \ENDFOR
\STATE \textbf{end procedure}
\end{algorithmic}
\end{algorithm}

The one-time conversion cost is $O(|D|w)$. After conversion, counting uses vectorized bitwise intersections. Space is $O(|I_f||D|)$ bits for frequent item bitsets, where $|I_f|$ is the number of frequent singleton items.

\section{Optimization Strategies}
\subsection{Optimization 1: Vertical Tidsets and Bitsets}
Claude Opus' code review identified repeated support counting and redundant transaction scans as the major Apriori bottlenecks. The first optimization converts each item into a transaction-id set or Boolean bitset. This removes repeated horizontal scans after initialization and lets support counting use intersections rather than nested transaction-candidate subset checks. The trade-off is additional memory for vertical structures, but dense benchmark data benefits because many intersections reuse the same compact representation.

\subsection{Optimization 2: Parallel Tensor Support Counting}
The second optimization applies the same representation to tensor hardware. Boolean arrays are stored as PyTorch tensors when CUDA is available, so support counting becomes \texttt{bitwise\_and} followed by \texttt{sum}. This is more hardware-friendly than Python loops. Claude also suggested warmup runs, garbage collection before measurements, support-ascending ordering, and recording candidate counts; these suggestions are reflected in the benchmark script and analysis tables.

\section{Discussion}
The main lesson is that Apriori's pruning property is necessary but not sufficient for dense data. When many itemsets survive pruning, the cost of checking candidates dominates. Vertical representation changes the unit of computation: support is no longer found by asking whether a candidate is contained in each transaction; it is found by intersecting occurrence vectors.

Tensor-Eclat has trade-offs. It pays a one-time conversion cost and may be slower for very small datasets or very high support thresholds where there is little mining work. GPU execution also depends on available VRAM and PyTorch/CUDA installation. The NumPy fallback keeps the implementation portable, but the largest speedups require tensor acceleration.

The Claude-assisted review was most useful in identifying engineering details that affect reproducibility: run multiple trials, include a warmup pass for CUDA context initialization, track memory and candidates, free older tidsets where possible, and sort items by support before recursive Eclat expansion. These suggestions turn the project from a pair of algorithms into a measurable experimental pipeline.

\section{Conclusion}
This project compared classical Apriori with a contemporary BitTable/Tensor-Eclat direction for frequent itemset mining. The optimized approach replaces repeated database scans with vertical bitsets and maps support counting to vectorized tensor intersections. Across the reported benchmark artifact, the optimized method reaches up to 108$\times$ speedup and substantially lower memory use. Apriori remains an excellent conceptual baseline for explaining anti-monotonic pruning, but modern FIM performance depends heavily on representation, locality, and parallel execution. Future work should run the pipeline on the full downloaded FIMI datasets, compare against FP-Growth and Spark implementations, and add multi-GPU or distributed sharding for larger transaction collections.
